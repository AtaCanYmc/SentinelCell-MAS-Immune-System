# SentinelCell: Container Policy

## Overview
This policy defines the security and resource boundaries for SentinelCell agents running within containerized environments (e.g., Docker, Kubernetes). The goal is to enforce the **Principle of Least Privilege** for all agentic processes.

## 1. Resource Constraints
To prevent "Runaway Agent" scenarios where an LLM agent consumes all system resources:
- **CPU Limit:** Each agent container is restricted to 0.5 vCPU.
- **Memory Limit:** Memory usage is capped at 512MB per agent.
- **Disk IO:** Write access is restricted to the `/logs` and `/temp` volumes only.

## 2. Network Security
- **Egress Filtering:** Agents are prohibited from initiating outbound connections to any address except the pre-authorized MCP Registry URL and the configured LLM Provider API endpoint.
- **No-Internal-Access:** Agents have no access to the host's `/var/run/docker.sock` or host network interfaces.

## 3. Environment Isolation
- **Secrets Management:** Environment variables (e.g., `GEMINI_API_KEY`) must be injected via read-only secrets. They are never hardcoded or stored in the image.
- **Ephemeral Filesystem:** All agent containers must run with a read-only root filesystem, except for designated persistent data directories.

## 4. Sandbox Enforcement
- Every repair logic generated by the agent is executed within an ephemeral "Sidecar" container that is destroyed immediately after the `Antigravity` sandbox validation confirms the output integrity.

## 5. Compliance
- Any container that attempts to violate these policies will be automatically killed by the `SentinelGuard` process, and an entry will be generated in `logs/agent_decisions.json`.
